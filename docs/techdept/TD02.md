## Judgment

You need a **central error strategy**, not a single central error function.

Right now the repo already has:

* a top-level `AppErrorBoundary`
* a tiny observability buffer
* many local `catch { return null }` / fallback branches

That means crashes are only centralized at the very top, while **real failures are being hidden lower in the stack**.

So the current problem is not “missing one central handler”.
The real problem is: **errors have no model**.

---

## What is wrong now

Examples of the current pattern:

* storage parse/migration fails → `return null`
* obstacle mesh generation fails → `return null`
* worker dispatch fails → silently fall back to main thread
* share/decode failures are partially swallowed

That is not error handling.
That is **error erasure**.

If you now add one `handleError(error)` somewhere central, nothing important changes.
You will still lose:

* what failed
* where it failed
* whether the app is degraded
* whether the user can recover
* whether this is expected or a bug

---

## What you should do instead

## 1. Define error classes or error codes by boundary

Do not throw raw `Error` everywhere.

Use a small set of explicit app-level errors, for example:

```ts
type AppErrorCode =
  | 'STORAGE_CORRUPTED'
  | 'SHARE_PAYLOAD_INVALID'
  | 'GEOMETRY_BUILD_FAILED'
  | 'SOLVER_FAILED'
  | 'HEATMAP_WORKER_UNAVAILABLE'
  | 'MAP_INIT_FAILED'
  | 'PLACE_SEARCH_FAILED'
```

Each error should carry:

* `code`
* `message`
* `cause`
* `context`
* `severity`
* `recoverable: boolean`

---

## 2. Use `Result<T, E>` at important boundaries

For expected failures, stop throwing and stop returning `null`.

Bad:

```ts
generateObstacleMesh(...): RoofMeshData | null
```

Better:

```ts
type Result<T, E> =
  | { ok: true; value: T }
  | { ok: false; error: E }
```

Then:

```ts
generateObstacleMesh(...): Result<RoofMeshData, GeometryBuildError>
readStorage(...): Result<ProjectState, StorageReadError>
```

This is the main fix.

---

## 3. Split handling by layer

### Domain / analysis layer

No UI here.
Return typed errors.

### Application layer

Decide what to do:

* stop current action
* degrade feature
* retry
* use fallback

### UI layer

Decide how to present it:

* toast
* inline warning
* degraded mode banner
* silent dev log only

### Global boundary

Catch only truly unexpected crashes.

That means:

* `AppErrorBoundary` stays, but only as last resort
* it must not be your main error system

---

## 4. Add one central reporter, not one central decision-maker

You do want one place to **record** errors.

For example:

```ts
interface ErrorReporter {
  report(error: AppError, context?: Record<string, unknown>): void
}
```

This should be called from all layers.

But that reporter should not decide product behavior.
It should just:

* log
* emit telemetry
* attach metadata
* maybe dedupe

So yes to **central reporting**.
No to **central swallowing**.

---

## 5. Distinguish expected failure from programmer bug

This is critical.

### Expected failure

Examples:

* corrupted shared payload
* place search API timeout
* worker not available

These should become typed operational errors.

### Programmer bug

Examples:

* impossible state
* invalid geometry assumptions
* broken reducer invariant

These should surface hard:

* throw
* hit boundary
* be reported loudly

Right now the repo mixes both and turns too much into “best effort”.

---

## What I would implement

## Phase 1 — minimal useful version

1. introduce `AppError` shape
2. introduce `Result<T, E>`
3. replace `catch { return null }` in critical paths
4. add one `reportAppError()` service
5. mark degraded feature states explicitly

## Phase 2

6. convert worker/storage/share/geometry boundaries to typed errors
7. show proper inline UI states instead of silent fallback
8. make “degraded mode” explicit per feature, not only app-wide

## Phase 3

9. add severity and recovery policy:

   * `info`
   * `warning`
   * `error`
   * `fatal`

---

## Concrete examples for this repo

### Storage

Current behavior: bad payload → `null`
Better:

* return `STORAGE_CORRUPTED`
* show “Saved project could not be loaded”
* offer reset/import raw payload

### Obstacle mesh

Current behavior: generation fails → `null`
Better:

* return `GEOMETRY_BUILD_FAILED`
* keep obstacle visible as invalid object marker
* log exact obstacle id + shape + input dimensions

### Worker

Current behavior: fail → fallback silently
Better:

* fallback is allowed
* but emit `HEATMAP_WORKER_UNAVAILABLE`
* mark feature as degraded
* make it visible in dev/debug UI

---

## Final judgment

**Yes, do it.**
But do **not** build a fake “central error handler” that just catches and logs everything.

That would be more generator architecture.

What you want is:

* **typed errors**
* **Result-based boundaries**
* **central reporting**
* **local handling**
* **global boundary only for true crashes**

That is real error architecture.
