# Refactoring Plan: Extract External HTTP Clients into `src/app/clients`

## Objective

Refactor the application so that all communication with external HTTP services is isolated in a single, clear location: `src/app/clients`.

The goal is **not** to introduce a new architectural layer or add unnecessary abstractions. The goal is only to ensure that:

* external endpoints are defined in one place,
* all raw HTTP calls are centralized,
* feature modules no longer contain direct `fetch` logic,
* HTTP clients remain simple and contain no business logic.

---

## Target Directory Structure

```text
src/
  app/
    clients/
      photonClient.ts
      openMeteoClient.ts

    features/
      place-search/
        providers/
          photonPlaceSearchProvider.ts
      sun-tools/
        forecast/
          openMeteoForecast.ts
```

---

## Scope

This refactor should cover all code that:

* knows an external service URL,
* builds HTTP query parameters,
* performs `fetch`,
* validates HTTP response status,
* parses raw JSON response bodies.

Based on the current codebase, the relevant external integrations are:

1. **Photon API**
   Used for place search.

2. **Open-Meteo API**
   Used for forecast / irradiance data.

These are the only clear HTTP integrations that should be extracted at this stage. Static URLs such as tile server constants should not be moved unless they become actual HTTP client modules.

---

## Design Rules

### 1. Clients must stay simple

Each client should only contain:

* base URL or endpoint definition,
* request parameter construction,
* HTTP call execution,
* response status validation,
* raw JSON parsing.

### 2. No business logic inside clients

Clients must **not** contain:

* caching,
* retry policies,
* observability orchestration,
* domain mapping,
* UI-facing transformations,
* toast/error presentation,
* orchestration across multiple requests.

### 3. Feature modules remain responsible for application logic

Feature modules should continue to own:

* mapping external payloads into app-specific models,
* caching and retry where already needed,
* UI state handling,
* orchestration logic,
* domain-specific parsing and calculations.

---

## Planned Changes

## Phase 1 — Introduce `src/app/clients`

Create a new directory:

```text
src/app/clients/
```

Add the first two clients:

* `photonClient.ts`
* `openMeteoClient.ts`

---

## Phase 2 — Extract Photon HTTP logic

### New file

`src/app/clients/photonClient.ts`

### Responsibilities

* define Photon endpoint,
* build search query parameters,
* execute HTTP request,
* validate response,
* return raw Photon payload.

### Example responsibility boundary

The client should do something equivalent to:

```ts
searchPlaces(args): Promise<PhotonResponse>
```

### Update existing feature code

Refactor:

`src/app/features/place-search/providers/photonPlaceSearchProvider.ts`

So that it:

* calls `photonClient`,
* maps raw API response into `PlaceSearchResult[]`,
* keeps feature-specific behavior outside the client.

### What stays outside the client

* mapping to app models,
* local search behavior,
* debounce,
* UI cache,
* feature-level observability.

---

## Phase 3 — Extract Open-Meteo HTTP logic

### New file

`src/app/clients/openMeteoClient.ts`

### Responsibilities

* define Open-Meteo endpoint,
* build request query parameters,
* execute HTTP request,
* validate response,
* return raw forecast payload.

### Example responsibility boundary

The client should do something equivalent to:

```ts
getForecast(args): Promise<OpenMeteoForecastResponse>
```

### Update existing feature code

Refactor:

`src/app/features/sun-tools/forecast/openMeteoForecast.ts`

So that it:

* prepares domain-specific parameters,
* calls `openMeteoClient`,
* parses raw response into app-specific forecast structures,
* keeps retry/cache/orchestration outside the client.

### What stays outside the client

* cache,
* retry,
* domain transformations,
* irradiance-specific parsing,
* multi-roof orchestration,
* partial failure handling.

---

## Phase 4 — Enforce the boundary

After both clients are introduced:

1. Search the repository for all `fetch(` usages.
2. Verify that direct HTTP calls exist only inside `src/app/clients`.
3. Remove any remaining duplicated endpoint/request logic from feature modules.
4. Confirm that feature modules depend on clients rather than external URLs directly.

---

## Acceptance Criteria

The refactor is complete when all of the following are true:

* All direct communication with external HTTP services is located in `src/app/clients`.
* Feature modules no longer contain raw `fetch` logic for external services.
* Clients are thin and contain no business/domain logic.
* Existing behavior remains unchanged.
* The codebase has a single obvious place to find and update external integrations.

---

## Explicit Non-Goals

This refactor must **not** introduce:

* a shared `BaseHttpClient`,
* factory patterns,
* interceptors,
* repository/service/gateway layering,
* generic integration abstractions,
* unnecessary normalization of unrelated external resources.

This codebase currently does not justify that complexity. The correct solution is a small structural cleanup, not a framework.

---

## Recommended Implementation Order

1. Create `src/app/clients/photonClient.ts`
2. Refactor `photonPlaceSearchProvider.ts` to use the new client
3. Verify place search behavior
4. Create `src/app/clients/openMeteoClient.ts`
5. Refactor `openMeteoForecast.ts` to use the new client
6. Verify forecast behavior
7. Search for remaining `fetch(` usage and clean up any leftovers

---

## Expected Outcome

After this refactor:

* external integrations will be easy to locate,
* feature files will be smaller and cleaner,
* HTTP responsibilities will be clearly separated,
* the codebase will improve without introducing architectural overhead.

This is a focused structural refactor, not a redesign. That is the correct level of change for the current application.
