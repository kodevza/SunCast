Yes. This is the right final split.

`map-editor` should own exactly **three sub-features**:

* **DrawTools** — everything related to drawing polygons
* **MapView** — base map runtime and interaction
* **MapObjects** — rendering of generated 3D roof / obstacle objects

The current issue is simple: `DrawTools` is too thin, and `MapView` is too fat.

# Refactor goal

Restructure `map-editor` so responsibilities are explicit:

* **DrawTools** handles drawing workflow
* **MapView** handles the map itself
* **MapObjects** handles rendered 3D geometry and roof layers

This is not a sidebar refactor.
This is a **map-editor internal decomposition**.

# Responsibility split

## 1. DrawTools

Owns all drawing-related behavior.

This should include:

* draw mode selection: roof / obstacle
* start drawing
* cancel drawing
* undo last point
* finish polygon
* drawing status text
* drawing shortcuts
* future drawing controls:

  * snap options
  * constrained length
  * guided angle
  * close polygon action
  * clear draft
  * draw help / hints

Today `DrawTools.tsx` is mostly one mode toggle plus a few buttons.
That is not a real feature yet. It is just a panel fragment.

## 2. MapView

Owns only the base map runtime.

This should include:

* MapLibre lifecycle
* map container
* style creation
* source registration
* map navigation
* orbit camera
* sun perspective
* click / drag / hit-testing
* draft preview on map
* selection interactions

It should **not** know how 3D roof layers are built internally.

## 3. MapObjects

Owns only rendered map objects.

This should include:

* roof mesh rendering
* obstacle mesh rendering
* heatmap overlay rendering
* custom 3D layers
* geometry adaptation for renderer
* layer visibility sync
* Three.js or custom renderer internals

This is the place for the current `roof-layer/*` logic.

# Required design rule

`MapView` may decide **when** objects should be updated.

`MapObjects` decides **how** they are rendered.

That is the boundary.

So:

* `MapView` passes map instance + object data
* `MapObjects` owns layer refs and rendering internals
* `DrawTools` never touches renderer internals

# Main code change

Today `MapView.tsx` directly knows about:

* `roofLayerRef`
* `obstacleLayerRef`
* `heatmapLayerRef`
* `useMapLayerSync(...)`

That is wrong after the refactor.

It should become something like:

```ts
useMapObjectsSync({
  mapRef,
  mapLoaded,
  roofMeshes,
  obstacleMeshes,
  heatmapFeatures: shadingHeatmapFeatures,
  orbitEnabled,
  meshesVisible,
  shadingEnabled,
  shadingComputeState,
})
```

And the layer refs should be owned inside `MapObjects`, not in `MapView`.

# Implementation plan

## Phase 1 — establish the three sub-features

* keep `map-editor` as parent
* create `MapObjects/`
* expand `DrawTools/` beyond one component
* do not change behavior yet

## Phase 2 — move rendering code out of `MapView`

* move `roof-layer/*` to `MapObjects/`
* rename `useMapLayerSync` to `useMapObjectsSync`
* isolate all 3D layer sync there

## Phase 3 — harden `DrawTools`

* move drawing-only actions and keyboard logic into `DrawTools`
* keep it as the single place for polygon drawing UX
* prepare it for future drawing features

## Phase 4 — simplify `MapView`

* remove direct knowledge of mesh layer implementation
* keep only map runtime, interactions, sources, and camera

# Acceptance criteria

The refactor is done only if:

* `DrawTools` owns all polygon drawing workflow
* `MapView` owns only map runtime and interaction
* `MapObjects` owns all rendered roof / obstacle / heatmap layers
* `MapView` no longer imports `roof-layer/*`
* `DrawTools` is no longer just a button wrapper
* future drawing features can be added without touching `MapObjects`
* future rendering changes can be made without touching `DrawTools`

# Final judgment

This is the correct architecture cut.

The previous state was muddled:

* `DrawTools` too weak
* `MapView` overloaded
* object rendering buried inside map code

The clean decomposition is:

**DrawTools + MapView + MapObjects**
